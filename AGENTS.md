# AGENTS.md

本文件用于指导后续 Codex 在本仓库中的开发工作。除非子目录中存在更具体的 `AGENTS.md`，本文件约束整个仓库。

## 项目背景

- 本项目是基于 yudao-cloud / RuoYi 风格的新项目。
- 后端技术栈以 Java、Spring Boot 3.x、MyBatis Plus 为主。
- 新增 AI 知识库模块时，模块名固定为 `yudao-module-ai`。
- AI 能力接入优先遵循 Spring AI 的抽象与项目既有封装方式。
- 所有新增或修改代码必须遵循现有项目结构、命名、分层、异常处理、返回体、权限控制、日志和测试风格。

## AI 知识库模块约定

- `yudao-module-ai` 应保持与现有 yudao 模块一致的目录结构和分层方式。
- Controller 只负责请求入口、权限、参数校验、响应封装，不编写复杂业务逻辑。
- Service 负责业务编排、事务边界、外部 AI/向量库调用协调和核心流程控制。
- Mapper 只负责数据库访问，不承载业务判断、外部调用或复杂编排。
- 数据对象、VO、DTO、DO、Convert、Enum 等命名与放置位置应参考现有模块。
- 第一阶段向量库使用 PostgreSQL + pgvector。
- 需要在设计上预留 Qdrant 适配能力，避免把 pgvector 实现硬编码进业务编排层。

## 安全与配置

- 不允许在代码、测试、配置样例或文档中硬编码 API Key、密码、Token、私钥等敏感信息。
- 敏感配置必须通过环境变量、配置中心、部署配置或本项目既有安全配置机制注入。
- 提交配置样例时只能使用占位符，不得出现真实凭据。
- 日志不得输出完整密钥、Token、密码、用户隐私数据或大段原始请求内容。

## 接口与校验

- 所有新增接口必须具备基础参数校验。
- 请求参数优先使用项目既有校验方式，例如 Bean Validation 注解和统一异常处理。
- Controller 中不得绕过统一响应、统一异常、权限或租户机制。
- 新增管理端或用户端接口时，需参考现有模块的 API 路径、权限标识和菜单权限命名风格。

## 外部调用与日志

- 所有外部调用，包括大模型、Embedding、向量库、对象存储、第三方 HTTP API，都必须有异常处理和必要日志。
- 日志应包含排障所需的上下文，例如 tenantId、knowledgeBaseId、业务对象 ID、调用类型，但不得泄露敏感信息。
- 外部调用失败时，应转换为项目统一异常或业务可理解的错误，不得把底层异常直接暴露给前端。
- 对可能耗时或不稳定的外部调用，应考虑超时、重试、降级或异步处理，并遵循现有项目机制。

## RAG 与租户隔离

- RAG 检索必须包含 `tenantId` 和 `knowledgeBaseId` 过滤。
- 任何知识库文档、切片、向量、检索结果、会话或引用数据，都必须遵守租户隔离。
- 不允许跨租户检索、更新或删除知识库数据。
- 向量检索接口设计应避免与具体向量库强绑定，至少保留 PostgreSQL + pgvector 与未来 Qdrant 的实现替换空间。

## 数据访问

- MyBatis Plus 相关代码应遵循现有项目的 Mapper、DO、分页、Wrapper 和 XML 使用风格。
- Mapper 不处理复杂业务逻辑。
- SQL 或 Wrapper 条件涉及租户、知识库、删除状态、启用状态时，应显式保证过滤条件完整。
- 数据库迁移、初始化 SQL、索引和 pgvector 扩展相关脚本应遵循项目既有脚本目录与命名规则。

## 代码质量

- 优先复用项目已有工具类、异常类、枚举、Convert、分页对象、响应对象和测试基类。
- 不为单一场景创建过度抽象，但要避免把可预见的 Qdrant 适配点写死在业务层。
- 新增代码应保持职责清晰、方法长度适中、命名明确。
- 新增测试应覆盖主要成功路径、参数校验、租户/知识库过滤、外部调用失败等关键风险点。
- 修改现有代码前先阅读相邻实现，保持 yudao-cloud / RuoYi 风格一致。

## 每次交付要求

每次修改后，回复必须包含以下内容：

- 修改文件：列出新增或修改的文件。
- 运行命令：列出实际执行过的命令。
- 测试结果：说明测试、构建或检查结果；未运行时说明原因。
- 未完成事项：列出仍需处理的事项；没有则写“无”。
